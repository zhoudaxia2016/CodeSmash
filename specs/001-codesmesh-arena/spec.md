# 功能规格说明：CodeSmesh（大模型编程能力乱斗）

**功能分支**：`001-codesmesh-arena`  
**创建日期**：2026-04-02  
**状态**：草稿  
**输入**：用户需求摘要：构建名为「CodeSmesh」的模型编程能力测试系统（寓意「大模型大乱斗」）。核心：选一题、选两个模型并行测试，对比思路→代码→自测→系统用例通过率；记录通过率、自测完善性、运行时间。**首版由平台代调模型**（密钥在服务端，用户无需配置 API）；**后续迭代**再支持用户自带连接配置（如 API Key、Base URL 等）。配套：题目管理、测试集管理（由题目一键生成或手动添加）。体验上借鉴 [Arena 排行榜](https://arena.ai/leaderboard/) 的对比与发现感，但 CodeSmesh 必须能真实执行代码。非功能：模型生成代码在 QuickJS Wasm 沙箱中执行，单次超时 5 秒；前端为单页应用部署于 GitHub Pages；模型推理与排行榜等服务端能力由部署在 **Deno Deploy** 的 API 承担，且首版须具备基础限流与滥用防护。

## 范围与分期

- **首版（本规格主范围）**：用户从**平台提供的模型列表**中选择两个模型发起乱斗；所有供应商调用由**服务端代发**；用户浏览器**不接收、不存储**供应商 API Key。
- **后续迭代（明确延后）**：用户可配置自有模型连接（展示名、API Key、Base URL、模型标识等），配置策略与存储位置在专门迭代中定义；本版需求不得阻塞该扩展，但**不要求首版实现**。

## 用户场景与测试（必选）

### 用户故事 1 — 双模型同题乱斗（优先级：P1）

作为用户，我希望在测试页从题目列表中选一道题（如「两数之和」），从**平台提供的模型列表**中勾选两个模型，点击「开始测试」，系统并行经服务端调用两个模型，按固定阶段输出：**解题思路 → 可运行代码 → 自测用例与自测结论 → 在系统预设用例上运行的通过率**；并汇总展示便于并排对比的结果（含通过率、自测完善性、运行时间等维度）。

**为何此优先级**：这是产品核心价值与差异化（可执行、可对比），没有它则不成其为「乱斗」。

**可独立验证**：仅用内置示例题目与平台提供的至少两个模型即可完成一次完整乱斗并得到可核对的结果。

**验收场景**：

1. **假如** 已存在至少一道题目与平台提供的至少两个可选模型，**当** 用户选题、选两模型并启动测试，**则** 两路请求经服务端代调并行推进，界面能区分展示双方输出与最终对比指标。
2. **假如** 某一模型调用失败、超时或触发限流，**当** 测试进行中，**则** 该侧显示明确错误状态（含可操作提示，如稍后再试），另一侧仍可完成；整体可标记为「部分成功」并保留已得结果。
3. **假如** 模型生成的代码在沙箱中超时，**当** 执行预设用例，**则** 该次执行被终止并记为失败/超时，且不超过规定的最大执行时长。

---

### 用户故事 2 — 平台模型目录与公平使用（优先级：P2）

作为用户，我希望在界面上看到**当前可用的模型列表**（显示名、可选的简短说明或标签），无需也不应被要求填写供应商密钥即可开跑。作为运营方，系统必须能通过**限流与基础滥用防护**控制成本与公平性，避免公开入口被刷爆。

**为何此优先级**：平台代调模式下，模型目录是选型的前提；限流是上线必备，与代调同时存在才有可持续服务。

**可独立验证**：在未完成完整乱斗的情况下，仍能拉取/展示模型列表，并在模拟超限请求时得到一致、可理解的限流反馈。

**验收场景**：

1. **假如** 平台已配置若干模型，**当** 用户打开乱斗相关页面，**则** 至少能看到可选模型列表并能选择其中两个（若不足两个须明确提示不可用原因）。
2. **假如** 用户在短时间内的请求超过平台设定阈值，**当** 继续发起代调请求，**则** 请求被拒绝或排队策略生效，且用户看到**非技术恐吓式**的说明（如「请求过于频繁，请稍后重试」）。
3. **假如** 用户查看任意面向公众的界面或网络载荷，**当** 完成一次乱斗，**则** 不出现供应商 API Key 或等价机密（与 FR 一致）。

---

### 用户故事 3 — 题目与测试集管理（优先级：P3）

作为出题/维护者，我希望管理题目（题干、约束说明、函数签名/入口约定等使沙箱可测），并管理每题对应的**测试集**：支持从题目**一键生成**初始用例草稿，也支持**手动增删改**用例；测试集用于系统预设用例运行与通过率统计。

**为何此优先级**：支撑可持续扩展的题库与公平评测，但可在内置种子题库存在时后置实现。

**可独立验证**：在无模型调用情况下，仍能维护题目与用例并在沙箱中对手写参考代码跑通。

**验收场景**：

1. **假如** 用户新建题目并保存，**当** 在题目列表中查看，**则** 题目可被选中用于测试。
2. **假如** 用户点击「从题目生成测试用例」类操作，**当** 系统生成草稿，**则** 用户可继续手动调整后再用于正式评测。
3. **假如** 测试集为空或格式不合法，**当** 用户尝试启动需要预设用例的评测，**则** 系统阻止并提示补齐。

---

### 用户故事 4 — 排行榜与发现感（优先级：P4）

作为访客或参与者，我希望看到类似 Arena 的**排行榜/总览**：按题目或全局聚合展示各模型表现（如通过率、耗时等可公开指标），便于发现强模型与热门题目；数据经服务端 API 聚合（见假设）。

**为何此优先级**：增强传播与「爆品」属性，但不阻塞可用的乱斗 MVP。

**可独立验证**：前端展示来自 API 的聚合数据；无数据时展示空状态或说明。

**验收场景**：

1. **假如** 服务端存在公开排行榜数据，**当** 用户打开排行榜页，**则** 能看到排序列表与基础说明（指标含义）。
2. **假如** 用户选择不参与统计上报（若提供该选项），**当** 完成乱斗，**则** 不产生可识别个人的远程记录；默认策略见假设。

---

### 边界与异常

- 两模型输出格式极不规范、无法解析出代码块时，如何提示并允许用户重试或查看原文？
- 自测用例与官方用例冲突、或自测「全过」但官方用例失败时，如何在结论中明确标注「自测覆盖不足」？
- 网络中断、供应商 429、服务端鉴权失败或**平台限流**时，错误信息是否可操作（稍后重试、联系支持、换题等）？
- 用户只选了一个模型时，是否禁止开始或允许单模型基准跑？
- 题目要求的语言/导出形式与沙箱仅支持的能力不一致时，如何前置校验？

## 需求（必选）

### 功能需求

- **FR-001**：系统必须允许用户从题目列表选择**一道**编程题用于一次乱斗。
- **FR-002**：系统必须允许用户从**平台维护的可用模型列表**中选择**两个**模型发起乱斗（若仅选一个，行为见边界说明；默认建议禁止开始并提示）。
- **FR-003**：系统必须**并行**发起两路模型会话（从用户视角同时推进），并在界面并排或分栏展示进度与结果；实际供应商调用经**服务端代发**。
- **FR-004**：每路模型必须按约定阶段产出：**思路说明 → 代码 → 自测用例与自测执行/说明 → 基于系统预设用例的执行结果与通过率**；若某阶段模型未按格式返回，系统应降级展示原始文本并标明解析失败。
- **FR-005**：系统必须基于预设测试用例在受控沙箱中执行生成代码，并计算**通过率**；必须记录**端到端运行时间**（或分阶段耗时，至少包含可对比的总耗时）。
- **FR-006**：系统必须对**自测完善性**给出可复现的评判维度（例如：自测数量、是否覆盖边界、与官方用例结果一致性等），并在结果页展示；具体算法可在实现阶段细化，但须在规格层面可验收（见成功标准）。
- **FR-007**：系统必须提供**由平台维护**的模型目录（至少包含稳定标识与面向用户的显示名），供乱斗时选择；首版用户**不得**被要求提交供应商 API Key 才能使用核心流程。
- **FR-008**：对所有受支持的大模型供应商的 HTTP(S) 调用必须由**服务端**发起；**API Key 及等价机密仅保存在服务端安全配置中**，不得完整下发至浏览器或嵌入公开前端资源。
- **FR-009**：平台必须实施**限流与基础滥用防护**（例如按 IP、匿名会话标识或用户账号的频次/并发上限，具体策略可实现期确定），超限响应须向用户说明原因与建议（如稍后重试）。
- **FR-010**：系统必须提供题目的创建、编辑、列表与选用；题目内容须足以驱动沙箱评测（入口函数/IO 约定等）。
- **FR-011**：系统必须提供测试集管理：手动维护用例，并支持从题目**一键生成**测试用例草稿。
- **FR-012**：排行榜（若启用）必须仅依赖**非密钥**数据与聚合指标；展示需说明指标含义与数据来源限制。
- **FR-013（后续迭代，首版不验收）**：在后续版本中支持用户**自定义模型连接配置**（如 API Key、Base URL、模型标识等），与首版平台代调共存或作为高级模式；存储、安全与 CORS 等细节在该迭代规格中单独定义。

### 产品交付约束（已由产品方指定）

- **CON-001**：模型生成代码的执行环境必须为 **QuickJS Wasm 浏览器沙箱**，单次执行**硬超时 5 秒**；超时必须终止执行并计为失败。
- **CON-002**：前端交付形态为**单页应用**，默认托管于 **GitHub Pages**。
- **CON-003**：模型推理代调、排行榜等跨用户或持钥能力由部署在 **Deno Deploy** 的 API 提供；密钥与限流策略归属服务端责任范围。

### 关键实体

- **平台模型**：面向用户的显示名、内部稳定标识、服务端路由/供应商映射关系；**不含**用户个人密钥。
- **题目**：标题、描述、难度/标签（可选）、评测入口约定、参考说明（可选）。
- **测试用例**：所属题目、输入/期望输出或等价判定描述、是否启用、来源（手动/生成）。
- **乱斗会话**：题目、双方平台模型、各阶段产物、预设用例结果、自测评估、时间统计。
- **排行榜条目（若启用）**：模型标识、聚合指标、关联题目或全局维度；不含用户密钥。
- **用户模型配置（后续迭代）**：用户提供的连接参数；首版可不落库、可不实现。

## 成功标准（必选）

### 可度量结果

- **SC-001**：首次到访用户在**10 分钟内**（含阅读简短引导）能使用**内置示例**完成一次双模型乱斗并理解三项核心指标（通过率、自测完善性、耗时）的含义。
- **SC-002**：在标准网络下，**90%** 的乱斗会话能在用户可接受时间内完成界面反馈（具体上限与模型速度相关，以「无界面卡死、阶段状态可感知」为验收）。
- **SC-003**：对任意提交执行，沙箱均在 **5 秒内**结束或中断，不出现无限执行；抽检用例下**100%** 满足该时限。
- **SC-004**：在公开界面、前端可观测网络请求与默认上报载荷中**不得出现完整供应商 API Key**（人工抽检与设计审查通过）。
- **SC-005**：在刻意超过限流阈值的测试中，**100%** 触发可识别的限流响应，且用户可见文案**不含**内部堆栈或密钥片段。
- **SC-006**：用户调研或可用性测试中，**至少 80%** 参与者认为「对比两个模型的编程结果」比只看单一回答**更省时间**（适用于内测问卷，可选在首版后收集）。

## 假设

- 首版目标用户无需具备各供应商账号即可体验核心乱斗；后续 BYOK 面向进阶用户。
- 平台承担首版模型调用的**直接与间接成本**，因此限流与监控是上线前提而非可选项。
- 浏览器需支持 Wasm；前端与 Deno API 之间为同源或经文档约定的安全跨域方式通信（由实现期定义，不要求用户理解 CORS）。
- 排行榜为增强功能：**默认不上传**或需**明确勾选**；服务端仅存聚合与不可反推个人身份的数据（若收集）。
- 首版可提供**内置种子题库**，降低冷启动成本；「爆品」运营策略不在本规格范围，但界面预留排行榜与分享叙事空间。
- 「自测完善性」的量化规则由产品在实现时固定一套透明规则（版本化），并在 UI 中链接简短说明，避免黑盒。

## 借鉴 Arena 的体验要点（非约束）

- 清晰的**榜单/分类**与「一眼对比」布局，突出模型差异。
- 低门槛开始：预置内容与**极短路径**到达第一次对比（首版与**平台代调**一致）。
- 指标与**对战**叙事可提升参与感；CodeSmesh 以**可执行结果**为信任基础。
